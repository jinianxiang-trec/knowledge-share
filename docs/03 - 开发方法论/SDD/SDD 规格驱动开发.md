# **SDD 规格驱动开发**

当我们对 AI 说：“帮我做一个登录功能”，AI 花了几分钟产出一堆代码，最后很有信心地说：“完成了！这个功能已经可以使用了，完美！”

打开看一下，好像有登录表单、有验证逻辑、有错误处理，而且设计看起来十分专业。结果实际试一下，发现登录成功后没有跳转页面。怎么办？没关系，我们可以再跟 AI 说：“登录成功后要跳转到首页。”，AI 很快修好了，但这次忘记处理密码错误的情况。再继续修，它又改坏了另一个地方。就这样来来回回，原本以为十分钟能搞定的功能，搞了两个小时。

更惨的是，最后拿到的代码可能跟我们的预期架构完全不同。AI 用了我们从来没见过的包，写了一堆看不懂的抽象层。最后可能确实能运行，但你完全不知道它在干什么，也不知道以后怎么维护。

这不是你的问题，也不是 AI 的问题，这是方法论的问题。

## **Vibe Coding 的甜蜜与危险**

2025 年初，Andrej Karpathy 提出 Vibe Coding 的概念，就是“凭感觉写代码”，我们好像不用真的懂代码在干什么，只要跟 AI 聊天、许愿，它就会帮我们把东西做出来。做出来的东西有问题或是不喜欢？没关系，再继续跟 AI 聊天、许愿，它就会调整，整个过程就像在跟一个很厉害又超级有耐心的同事聊天，聊着聊着代码就写完了。

Vibe Coding 在某些场景下非常有效，例如想快速做个 POC 验证想法？没问题。想写个小工具自己用？完全可以。在探索一个新技术，不确定要怎么用？Vibe Coding 能让我们很快地摸到感觉。但当项目变大、需求变复杂或是团队成员变多的时候，Vibe Coding 的问题就会浮现。这也是 Vibe Coding 这个词最近好像已经变成一种嘲讽或是反面教材的原因之一。Vibe Coding 虽然很好玩，但可能会遇到一些问题...

第一个问题是 AI 产出的代码风格不一致。今天早上请它写会员登录功能，下午让它写购物车，明天再来一个订单管理功能，AI 可能会用三种完全不同的方法和架构。整个项目看起来像是三个不同的人写的，因为它确实是三次不同的对话产出的结果。

第二个问题是 AI 可能不会主动告诉你它漏掉了什么。我们跟 AI 说要登录功能，它就做登录功能，但它不一定会问“登录失败要怎么处理？”、“密码有什么格式限制？”、“要不要实现‘记住我’的功能？”。这些细节没跟 AI 讲，AI 可能就不会做，或者反过来没讲但做了你目前可能还不需要的功能。等你发现的时候，已经写了一堆代码，要回头改就很痛苦，说不定整个砍掉再重来一次还比较快。

第三个问题是最麻烦的，就是 AI 很会说“我完成了”。Amazon Kiro 的首席工程师 Al Harris 在 2025 年的一场演讲中讲得很直白，他说 AI 很擅长说“我做完了，我很满意，你应该也很满意”之类的话。但实际上测试没通过、逻辑有漏洞、边界条件没处理，它都会轻描淡写地带过。“哦对，测试没通过，但那个测试很烦，我试了三次都不行，就先跳过了。”

这不是 AI 在骗你，这是 LLM 本质上的问题。LLM 擅长模式识别和文字生成，但不擅长理解我们心里真正想要什么。给它一个模糊的指令，它就会用“最常见”的模式来填补那些模糊的空间。问题是，我们的需求往往不是最常见的那种。

于是有人开始想有没有什么方法可以让 AI 更可靠一点？“规格驱动开发（Spec-Driven Development）”，也就是大家最近常听到的 SDD 就是其中一个答案。

## **SDD 是什么？**

大家别误会，不要以为 SDD 是什么新的发明。“先写规格再写代码”这个概念早就存在了，瀑布式开发时代就已经在做这件事，TDD、BDD 这些方法论也都强调“先定义预期结果，再写实现”的精神。这些做法在 AI Coding 兴起之后经历了一次复兴，加上一些 KOL 的推广或刻意炒作，SDD 突然就这样红了起来。

SDD 的核心想法其实不复杂，就是在写代码之前，先把规格写清楚。但这并不是要我们写一份冗长的规格文档然后丢给 AI，而是先把规格变成 AI 和人类之间的共同语言，一个可以持续演进的共识基础。根据 Amazon Kiro 团队的定义，SDD 的工作流程分成三个阶段，每个阶段都有对应的文档。

第一个是 Requirements，把需求写成“用户故事（User Story）”的格式，附上明确的“验收标准（Acceptance Criteria）”，放在 requirements.md 里。  
第二个是 Design，产出技术设计文档，包含架构图、数据模型等，放在 design.md 里。  
第三个是 Tasks，把工作拆成一个一个可追踪的小任务，放在 tasks.md 里。  
这三份文档会随着项目演进而更新，而且规格跟代码要保持同步。

换句话说，SDD 试图解决的问题是当我们跟 AI 许愿“做一个登录功能”的时候，要怎么保证我们跟 AI 对于“登录功能”的理解是一致的，或者如何确保最后产出的代码真的做到了我们想要的事情。来细看 SDD 的三个阶段...

### **SDD 的三个阶段**

**第一阶段 Requirements**  
需求阶段。这个阶段的目标是把你脑袋里模糊的想法变成清晰的、可被验证的需求。这不只是简单写一句“要有登录功能”就好了，而是要写成用户故事的格式，然后附上具体的验收标准。例如：

作为一个用户，我希望能够用邮箱和密码登录，这样我就能访问我的个人资料。

这是用户故事，遵循“作为《角色》，我希望《功能》，这样我就能《好处》”的格式。然后验收标准可能会写成：

- “当用户输入正确的邮箱和密码的时候，系统应该将用户导向首页。”
- “当密码错误超过三次的时候，系统应该锁定账号三十分钟。”  
    这种“当...的时候，系统应该...”的写法其实就是下面会介绍的 EARS 格式。如果各位有写过 BDD 测试，可能会觉得这跟 Given、When、Then 的写法很像。两者的概念确实相近，只是 EARS 省略了 Given（前提条件），直接从触发条件开始描述。在这个阶段结束的时候，我们和 AI 对于“什么叫做完成”应该有一致的理解。

**第二阶段是 Design**  
设计阶段。有了清晰且一致的需求之后，接着由 AI 帮忙产出一份技术设计文档。这份文档会包括系统架构图、数据流程、数据模型、错误处理策略、测试策略等等。

设计阶段的重点是让我们有机会在写代码之前就发现问题。比如说，你可能会发现某两个需求是冲突的，或是某个技术选择会让后续的扩展变得困难。这些问题在设计阶段发现，修改成本很低。如果等到代码都写完了才发现，那就有点痛苦了。在这个阶段，我们可以要求 AI 加入线框草图，不需要用 Photoshop 或 Figma 这种专业的软件画，用 ASCII 画出大概的界面就行了。这样就能在写代码之前就确认界面的设计是我们想要的。如果不是，现在改规格很简单，总比等代码写完再改容易太多了。

**第三阶段 Tasks**  
任务阶段。在这个阶段，根据需求和设计，把工作拆成一个一个的小任务，每个任务同样都有明确的目标和验收标准，而且会追溯到最初的需求编号。

任务的“粒度（granularity）”很重要，不过这没有一定要切多大多小的标准答案。一般来说每个任务应该要足够小，小到“你”可以在一个合理的时间内完成并验证。为什么我特别标记“你”？因为每个人的经验能力不同，对任务大小的感受也会不同。你可以根据自己的经验来调整任务的大小，确保每个任务都是你能够掌握的范围。

这样做的好处是，如果某个任务出了问题我们很快就能发现，而且修复的范围是有限的。不会像 Vibe Coding 那样，做到一半才发现整个架构都错了，要全部重来。

### **EARS 让需求变得可执行**

刚才提到 EARS 格式，这是 SDD 里面一个很重要的概念。EARS 的全名是 Easy Approach to Requirements Syntax，翻译成中文大概是“需求语法的简易方法”。这个格式是 Alistair Mavin 和 Rolls-Royce 团队在 2009 年提出的，最初是为了分析喷气发动机控制系统的适航法规。他们发现需求最容易阅读的方式，是让各个子句永远按照固定顺序出现，然后逐渐精炼成这套格式。后来被空客、NASA、西门子等公司采用，也被很多大学纳入教材。

EARS 的核心是用“应”（shall）搭配固定顺序的条款（例如 当...应...），必要时用“则”（Then）拆出结果，以结果来说它是人类可以读懂的文本，又有足够的结构让机器可以解析。

举个例子：

```
当用户输入正确的邮箱和密码时
则系统应将用户重定向到首页
```

这个需求很清楚地说明了触发条件（输入正确的账号密码）和预期行为（跳转到首页）。没有模糊空间，没有需要猜测的地方。EARS 格式的好处有几个：

首先它强迫我们把需求想清楚。我们不能再写“登录要顺畅”这种模糊的东西，必须明确定义什么情况下应该发生什么事。

其次它让需求更容易被测试。因为需求被写成明确的触发条件与预期结果，我们更容易把它们整理成测试用例，进而验证行为是否符合规格。

最后它减少了 AI 的猜测空间。当使用结构化的语言描述需求，AI 就不需要去猜到底想要什么。它可以直接根据需求来生成代码，而且生成的结果更可预测。

### <span style="color: rgb(132, 63, 161);">SDD 的三个层级</span>

如果你去看目前市面上的几种 SDD 工具，你会发现它们对“Spec”的使用方式不太一样。在 Martin Fowler 团队的文章里提到可以把 SDD 分成三个层级：

**第一层是 Spec-first，规格优先。**  
这是最基本的 SDD 做法，就是在写代码之前先写好规格，然后用规格来引导 AI 生成代码。当这个任务完成之后，规格可能就被丢掉了，下次有新任务再写新的规格。大部分的 SDD 工具都至少支持 Spec-first，这已经比 Vibe Coding 好很多了，因为你至少有一份文档记录你当初想要什么。

**第二层是 Spec-anchored，规格锚定。**  
这个层级的做法是把规格保留下来而且还加进版本控制里，让它随着项目演进而更新，而且团队成员手上都有一份。当要修改功能的时候，我们不是直接去改代码，而是先改规格，然后让代码跟着规格走。Spec-anchored 的好处是永远有一份文档知道系统应该做什么。这对团队协作特别有帮助，因为新来的人可以先看规格，就能理解系统的设计意图，而不是只能从代码去猜测。

**第三层是 Spec-as-source，规格即源代码。**  
这是最激进的等级，我们根本不直接编辑代码，只编辑规格。代码完全由规格自动生成，甚至会在文件开头加上“从规格生成 - 请勿编辑”这样的警告。

对原本就会写代码的工程师来说，Spec-as-source 听起来有点梦幻，但它的逻辑其实很清楚。如果我们相信“规格是真相的来源”，那为什么还要让人去碰代码呢？让代码完全由规格生成，就能确保两者永远一致。

目前大部分的工具还是停留在 Spec-first 或 Spec-anchored 的阶段。Spec-as-source 还在实验阶段，这对规格的撰写能力要求很高，因为我们得要把规格写得非常非常精确，才能确保生成的代码是正确的。

### **工具比较**

市面上已经有几个工具在做 SDD 相关的事情。让我们来看看几个比较有代表性的工具。

<span style="color: rgb(186, 55, 42);">Kiro</span> 
Amazon Kiro 是从 VS Code 项目 fork 出来的，由 Amazon 团队开发。它的工作流程是 Requirements → Design → Tasks，三个阶段都有对应的 Markdown 文件。Kiro 的特色是它使用 EARS 格式来撰写需求，而且支持基于属性的测试，可以自动验证代码是否符合需求。

Kiro 还有一个叫做 Hooks 的功能，可以在文件变更的时候自动触发某些动作。比如说，你可以设定一个 Hook，让每次 CSS 文件变更的时候，自动检查它是否符合 Figma 设计稿。这对维护设计一致性很有帮助。

<span style="color: rgb(186, 55, 42);">Spec Kit</span>  
Spec Kit 是 GitHub 推出的开源工具，可以搭配 GitHub Copilot、Claude Code、Gemini CLI 等各种 AI Coding 工具一起使用。它的工作流程是 Constitution → Specify → Plan → Tasks，多了一个 Constitution 的概念，用来定义团队的基本原则和约束。

Spec Kit 的特色是高度可定制化。因为所有的东西都是放在项目目录里，我们可以自己修改模板、调整流程、加入额外的检查点。这对有特殊需求的团队来说很有弹性。不过根据 Martin Fowler 团队的观察，Spec Kit 目前还是偏向 Spec-first 层级，对于 Spec-anchored 的支持还不是很明确。每次建立规格都会开一个新的分支，这暗示他们把规格视为某次变更的产物，而不是功能的长期文档。

<span style="color: rgb(186, 55, 42);">Tessl Framework</span> 
Tessl Framework 这个我还没认真用过，不过从文档看起来它的愿景很有趣。Tessl 明确追求 Spec-as-source 的层级，希望让规格变成唯一需要维护的东西，代码完全由规格生成。

Tessl 的 Framework 会帮我们建立三种东西：Plans（行动计划）、Specs（意图描述）、Tests（验证测试）。除了 Framework 之外，他们还提供一个免费的 Spec Registry，里面有超过一万个开源软件库的 usage specs。这些 usage specs 的用途是帮助 AI 代理正确使用库的 API，避免产生不存在的方法或版本混淆的问题。

Martin Fowler 团队在试用 Tessl 的时候发现，因为 LLM 的非确定性，同样的规格多次生成可能会产出不同的代码。这意味着你需要把规格写得非常精确，才能确保每次生成的结果是一致的。Spec-as-source 是一个有趣的挑战，也让我们可以好好思考一下“什么是足够好的规格”。

<span style="color: rgb(186, 55, 42);">**OpenSpec**</span>  
最后要介绍的是 OpenSpec，这是一个开源的轻量级工具。如果你觉得 Spec Kit 的流程太重，像是为了拧一颗螺丝要写十几页文档，OpenSpec 可能更适合你。

OpenSpec 的设计理念是 Brownfield-first，也就是说它不只适合从零开始的新项目，更适合在现有系统上做修改。它把“真相的来源”和“变更提案”分开管理：`openspec/specs/` 存放目前系统应该长什么样子，`openspec/changes/` 存放你想要做的变更。这样的分离让你可以清楚看到每次变更的范围，也让多人协作的时候不会互相干扰。

> **冷知识**  
> 软件圈常用 Greenfield 和 Brownfield 来区分“全新项目”和“既有系统”这两种情境，这个比喻来自土地开发或环境工程。Greenfield 指的是未开发的绿地，可以从零开始建；而 Brownfield 是已开发过的棕地，可能有旧工厂要拆、土地被污染过，得先处理既有的东西才能动工。

工作流程很简单，先写一个提案说明你想做什么，跟 AI 讨论到双方都同意，然后就开始应用实现，做完之后就归档，把变更合并回 specs。整个过程不需要 API 密钥，所有东西都是 Markdown 文件，就放在项目目录里。

OpenSpec 支持的工具很多，Claude Code、Cursor、GitHub Copilot、Windsurf、Gemini CLI 等主流的 AI coding 工具都有支持。我目前主要的开发都是用 Claude Code 搭配 OpenSpec，用起来挺开心的，我也写了一篇文章分享心得，有兴趣可试用看看再跟我说用的手感如何。

### **SDD 不是万能药**

讲了这么多 SDD 的好处，我们也要诚实面对它的局限性...

第一个问题是 **SDD 会让开发速度变慢**。有开发者在 Hacker News 上分享他用 Kiro 的经验。他想做一个很简单的工具，显示 macOS 的全局键盘快捷键。他给了 Kiro 一份简短的规格和一些 TypeScript 的 schema，结果 Kiro 写了大概 5,000 行代码，包含测试。代码可以跑，测试也都有通过，但 5,000 行对这么简单的任务来说太夸张了。他后来手动把它缩减到 800 行，功能都没少。

这说明 SDD 适合有一定复杂度的项目。如果你只是要做一个小工具，直接 Vibe Coding 可能更有效率。

第二个问题是 **SDD 会改变工程师的角色**。有开发者说，用 SDD 开发流程的感觉不像在写代码，反而比较像在当产品经理。会变成花很多时间在定义需求、审核设计、验收结果，真正写代码的时间反而变少了，甚至没有什么代码是自己写的，感觉像是在监工 AI 的工作。

这对有些人来说是好事，对有些人来说是坏事。像我原本是很享受写代码的过程，SDD 就让我觉得好像少了什么。但如果各位更在意的是把事情做好，SDD 确实能帮我们更可靠地达成目标。

第三个问题是 **要做好 SDD 需要学习新的技能**。写好规格不是一件容易的事，这跟写代码一样需要训练。我们可能得学会用像是 EARS 格式之类的写法、学会怎么把模糊的需求变成清晰的验收标准、学会怎么设计可测试的系统。这些技能跟写代码不太一样，但都需要时间训练。

第四个问题是 **工具还在快速演进中**。Kiro、Spec Kit、Tessl 或是 OpenSpec 这几套工具对 SDD 的实现方式跟愿景都不太一样，也许下个月又会冒出新的工具、新的方法论。现在投入大量时间学习某个特定工具，不能保证这个投资会有长期回报。不过这点我倒不是很在意，因为 SDD 的核心概念是“先写规格再写代码”，这个概念不会变，而且我本来就有 TDD/BDD 的开发习惯，这对我来说没什么太大的影响。工具只是帮助我们实践这个概念的手段，学会了 SDD 的思维方式，就算之后再换其它工具也不会太困难。

### **什么情况适合用 SDD？**

根据 Spec Kit 的文档，SDD 适合三种开发阶段：

第一种是 **全新项目（0-to-1 Development）**，从零开始建立一个系统。这时候花时间写规格是值得的，因为你有机会在最一开始就把事情想清楚，避免后面付出更大的代价去修正错误的设计决策。

第二种是 **探索性开发（Creative Exploration）**，尝试不同的实现方式。当你不确定要用什么技术、什么架构的时候，可以用 SDD 的方法同时探索多种可能性。因为规格是跟实现分开的，你可以用同一份规格去尝试不同的技术组合，看哪个最适合。

第三种是 **渐进式改进（Iterative Enhancement）**，在现有的代码上工作。这包括新增功能和旧系统现代化，可能也是 SDD 最能发挥价值的场景。现有系统通常有很多隐藏的约束和假设，如果你直接让 AI 去改代码，很容易改坏其他地方。先写规格、厘清新功能如何与现有系统互动，能大幅降低出错的概率。

再以 OpenSpec 为例，specs 和 changes 的分离设计就是针对既有系统这个痛点。specs 文件夹告诉 AI“这些是已经存在的规格，你不能乱动”，changes 文件夹则是“这次想做的变更”。更关键的是提案阶段强迫你先想清楚“这次变更会影响什么”，列出受影响的 specs 和受影响的代码。这个动作本身就是在厘清边界，如果你自己都列不出来，AI 更不可能知道。

不过既有系统有个前提，就是你得先有 specs 才可以锚定。如果现有系统本来就没有规格文档，你得先花时间把现有行为记录下来，这是导入成本。很多公司现有的项目都是“代码就是文档”的状态，要回头补 specs 需要一点技术和意志力。比较务实的做法是渐进式导入，先从新功能开始建立 specs，慢慢积累，而不是一开始就想把整个系统的规格都补齐。

### **小结**

回到文章一开始的那个问题，当 AI 说“完成了”，你怎么知道它真的完成了？我想答案应该是：

**在开始之前，先定义什么叫“完成”。**

这听起来很简单，但做起来不容易。这需要我们改变工作方式，从“边做边想”变成“想清楚再做”；这需要学习新的技能，把模糊的想法变成清晰的规格；这还需要接受前期导入会多花一些时间的事实，不过别担心，这些时间会在后期慢慢赚回来。

如果你现在主要是用 AI 来做小项目、写 POC、或是探索新技术，Vibe Coding 可能还是最有效率的方法。享受那个快速迭代、即时反馈的感觉。但如果你开始做更大的项目、跟团队协作、或是需要长期维护的系统，我会建议试试看 SDD。不一定要用特定的工具，光是养成“先写规格再写代码”的习惯，就能让你跟 AI 的协作品质提升很多。贴心小提醒，选择工具之前，一定要先确定目前自己的情况，不要别人说什么好用就跟风用...

SDD 不是要取代 AI，而是要**让 AI 的产出更可控、更符合我们的期待**，也就是说，与其完全相信 AI 说“我做完了”，不如用更明确的规格和自动化测试来把关。通过结构化的文档描述需求、明确的验收标准以及自动化的验证，我们希望可以把 AI 变成一个可以信任的协作伙伴。

下次 AI 再跟你说“完成了”的时候，也许可以反问它一句“**完成了什么？**” :)
