    Agent harness engineering

    最近、ハーネスエンジニアリングという言葉をよく耳にします。

    https://openai.com/index/harness-engineering/

    https://blog.langchain.com/improving-deep-agents-with-harness-engineering/

    https://x.com/naoya_ito/status/2027223346526708092

    現時点で自分が見かけたいくつかの記事について、内容と理解を整理しておきます。だいたい OpenAI のやつ 6 割、他 3 割、自分の意見 1 割という感じです。

    ハーネスエンジニアリングとは¶

    ハーネスエンジニアリングという名称を使い始めたのは、自分の観測範囲だと Mitchell Hashimoto 氏のブログに登場するのが最古かなと思います。 （概念自体は前からあったので、誰が言い始めたかは重要ではないですが）

    https://mitchellh.com/writing/my-ai-adoption-journey#step-5-engineer-the-harness

    I don't know if there is a broad industry-accepted term for this yet, but I've grown to calling this "harness engineering." It is the idea that anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again.

    エージェントがミスや想定外の振る舞いをした時に、再発防止をするというシンプルなアイデアです。AIにセルフレビューと修正を行わせることで、最初の出力からベストに近いものを出させたいというわけですね。

    Mr. Hashimoto の定義では、2つのやるべきことがあります。

    • AGENTS.md の改善: エージェントが誤った振る舞いをする場合は、AGENTS.md を更新して再発を防止する。
    • ツール整備: スクリーンショット/テストランナーなど、エージェントが使用できる専用のツールを整備する。

    これらを通して、エージェント自身が自らの作業を検証できるようにすることで、より長時間の自律的実行が可能になります。

    OpenAI の発信¶

    Hashimoto 氏の発信から程なくして、OpenAI がハーネスエンジニアリングについての記事を出しました。 実体験ベースで非常にわかりやすく書かれており、この記事を機に Harness Engineering という言葉が広まった印象があります（ただし本文中に harness という単語はほとんど出てきません笑）。

    https://openai.com/index/harness-engineering/

    ざっくり3行でまとめるとこんな感じです。

    • Codexだけで内部ベータのプロダクトをゼロから作り、社内のデイリーユーザーと社外アルファユーザーに提供している。
    • 5ヶ月でおおよそ 100 万行を、ゼロからエージェントコーディングのみで作った (no manually-written code)。
    • 重要なのはコーディングではなくエージェントのための土台作りとレバレッジ。

    以下で OpenAI のやったことと主張をまとめます（自分の解釈が若干入ります）。

    エンジニアの役割の再定義¶

    コーディングタスクが無くなったことで、エンジニアの役割は system, scaffolding, leverage にシフトしたとのこと。

    以下自分の用語解釈です。

    • system: 開発全体が安定して回る仕組み。CI/CD、マージゲート、テレメトリ基盤など。
    • scaffolding: エージェントのための足場づくり。リポジトリ構造、テンプレ、AGENTS.md、汎用抽象化、サンプル、ツールのスクリプト化など。
    • leverage: 人間の労力を AI でいかにスケールさせるか。自動化戦略や速度のためのリスクテイクなど。

    OpenAI チームはこれらを depth-first (深さ優先) で実施したと述べています。さらっと書かれてますが自分は結構重要な思考だと思っていて、

    • 深さ優先
      • 1つのユーザーストーリーを選択し、設計/実装/テストなどをフルで通す
      • その過程で生まれた部品や抽象や仕組みを用いて、更に大きなものを作っていく
    • 幅優先
      • 色々なユーザーストーリーを浅く作って全体像を完成させる
      • 並列度が上がるが、部品や抽象や仕組みが揃っていない分、同じところで詰まる

    という対比だと解釈しました。幅優先だと初動は速いが、再利用性が低い分長期的にはレバレッジが効きにくいのでエージェント相性が悪いという理解です。

    また、本プロジェクトにおけるエンジニアのシステム関与の大半はプロンプティングであり、レビューと修正のループをエージェント自身に回させるような流れになっています。

    1. エージェントがローカルでセルフレビュー
    2. ローカル/クラウドで、別のエージェントがレビュー
    3. 人間・エージェント双方からのフィードバックに返答・修正
    4. エージェントのレビュアーが満足するまで反復

    このイテレーションの中で必要なコンテキストについても、エージェント自身がツールを使って直接取得できるようにしているようです。 この営みにより、レビューの大部分をエージェント同士で完結させているようです。

    アプリケーションの把握可能性 (legibility) を高める¶

    実装のスループットが向上したことで人間の QA が律速になったため、エージェント自身に QA 能力を持たせて作業を移譲したと書かれています。

    UI の場合は、

    • 作業ごとに git worktree を切り、Codex が実際のアプリインスタンスを起動して検証できるようにした
    • Chrome DevTools MCP をエージェントの実行環境に埋め込み、アプリの画面を操作できるようにした
    • DOM スナップショット、スクショ、画面遷移等をスキル化した

    オブザーバビリティ の場合は、

    • git worktree ごとに、エフェメラルなオブザーバビリティスタックも立ち上げる
    • Codex は LogQL や PromQL といったクエリ言語を用いて、調査・改善を行う
    • 性能や信頼性についての指示が実行可能になる (e.g. 起動が800ms以内で終わることを保証して)

    といった調子で、worktree ごとに隔離されたアプリ+観測データに対して Codex が作業できるようにしていたとのこと。

    ちなみにここでいう legibility とは、「エージェントがシステムの挙動をどれだけ定量的に把握できるか」くらいのニュアンスで使用されているものと思われます。 「可読性」と訳されているのをよく見ますが、readability とは異なる概念なので注意して下さい。本記事では「把握可能性」と表現することとします。

    リポジトリナレッジを知識基盤 (system of record) として整備する¶

    OpenAI チームの、コンテキストマネジメントにおけるトライアンドエラーが書かれています。

    • 最初は巨大な一つの AGENTS.md を作るアプローチを取ったが、すぐに崩壊した
    • コンテキスト圧迫、過学習、重要性の分散、保守困難、自動リント困難...etc
    • 100 行程度の目次としての AGENTS.md と、知識基盤として構造化された docs/* に分けた
    AGENTS.md

    In-repository knowledge store layout

    docs/* には、

    • 設計系: エージェントファーストな core-beliefs.md (=開発手順や設計上の判断軸？) や、verification status (=設計docのドリフト状態？) を含む
    • 構造系: ARCHITECTURE.md のようなドメイン/パッケージ各層のトップレベルメンタルマップ
    • 品質系: ドメイン/パッケージ各層の品質のスコアリングとトラッキング
    • 計画系: 一定の規模の実行計画ドキュメント自体を第一級アーティファクトとして扱い、進捗と意思決定のログを残す。完了/未完了状態と技術負債までを含めてリポジトリにチェックインする。

    のドキュメントを含めており、AGENTS.md などのエンドポイントをインデックスマップとして小さく保つことでコンテキストをマネジメントしているとのこと。 またこれらのドキュメントが危殆化しないよう、専用のリンタ/CI/エージェントでリンク切れやドリフトを検知/修正 PR できるように自動化しているようです。

    エージェントの把握可能性 (legibility) がゴール¶

    AIの把握可能性を確保するために、ひたすらにリポジトリにコンテキストを詰め込み、よりレバレッジを効かせられるようにしたとのこと。

    • Google Docs や Slack での議論等は Codex にとっては把握不可能 = 存在しないに等しい
    • 場合によっては upstream の意図が把握できない外部ライブラリよりも、トレースできる内製化を優先した

    アーキテクチャとテイストを強制する¶

    ここもコンテキストマネジメントの話です。

    • エージェントには不変条件を矯正し、実装の細部を管理しない (e.g. 境界 (boundaries) でデータをパースすることを強制するが、パースライブラリは指定しない)。
    • エージェントは厳格な境界 (boundaries) と予測可能な型構造の上で上手く動作する ので、各ビジネスドメインのアーキテクチャ層を固定し、各層の依存の方向性を厳格に制御する。
    • これらの制約 (constraints) を専用のリンタとテストで機械的に判定する

    非常に厳格にレイヤードされた構造になっていますが、曰く、

    • 普段は後回しにするこの手の設計が、エージェント協業においては事前に満たすべき条件となる。この制約 (constraints) こそが崩壊・ドリフトせずに速度を出すために非常に重要。
    • 命名規則やファイルサイズ制限、構造化ログ、プラットフォームごとの信頼性要件(platform-specific reliability requirements)。
    • 例のごとく、この強制のためのカスタムリントとテストを用意している。カスタムなので、エージェントにプロンプトとしてそのまま注入できるようなエラーメッセージが書ける。

    とのこと。カスタムリンターにエージェントフレンドリーなエラーメッセージを実装するのはいいアイデアですね。

    またこの章では、制約 (constraints) が有効な範囲についても言及しています。

    • 優れた組織構造のように、境界 (boundaries) 付近においては厳格に制約をあたえ、各境界 (boundaries) の内部においては自由を与える。
    • エージェントの生成物が人間の好むスタイルになっていない可能性もあるが、動作が正しく、保守可能で把握可能 (legible) であれば気にしない。
    • レビューコメント、リファクタPR、バグといって追加のテイストは、ドキュメント化するかリンタに落とし込む。特に、ドキュメントが守られない場合にリンタに昇格させるのがいい。

    スループットがマージ哲学を変えてしまった¶

    エージェントのスループットが向上した結果、多くの開発規範が非生産的になってしまったと述べています。

    • マージゲートは最小化され、PRは短命になる。
    • フレーキーなテストは、原因究明するよりも再実行で通れば OK とする場合もある。
    • エージェントの登場により、実装は安く、修正のための待機は高くなった。

    実装スループットが低い環境では無責任な振る舞いであったことも認めており、損益分岐点が変わったという主張のようです。

    「エージェント生成 (agent-generated) 」が真に意味すること¶

    人間はいままでとは異なる抽象レベルで作業するようになるという主張です。

    • エージェントはコード/テスト/CI/リリース/ツール/ドキュメント/設計/ハーネスの評価/レビューと対応/スクリプト/ダッシュボード定義ファイル...を生成
    • 人間は、ユーザフィードバックを受け入れ条件に変換し、生成物を検証する役割にシフトする
    • エージェントが詰まっているときは、ツール/ガイドライン/ドキュメンテーションを整備させて障害を取り除く

    自律レベルの向上¶

    最近はエージェントが一つの新機能を E2E で駆動できるところまで仕上がってきたようで、単一のプロンプトに対して、

    • コードベースの現状の検証
    • 報告されたバグの再現
    • バグ再現のビデオキャプチャ
    • 修正の実装
    • アプリケーションを実行し、修正を検証
    • バグ修正のビデオキャプチャ
    • PR オープン
    • エージェントと人間のコメントに対応
    • ビルド失敗の検出と修正
    • 判断が必要な場合のみ人間にエスカレート
    • 変更のマージ

    をやってくれるらしいです。

    エントロピーの増大と GC¶

    Codex はリポジトリに既にあるパターンを参照して実装していくため、既存に微妙なやつが混ざってるとそのうちドリフトするという、AI駆動の負の側面について言及しています。

    • この手の AI Slop (= 低品質な AI 生成を指す) を掃除するために、最初は人力で週 1 日対応していたが、これはスケールしなかった。
    • 代わりに「ゴールデンプリンシプル (golden principles) 」を定義してリポジトリにチェックインし、機械的にチェックできるようにする。
    • 例えば「場当たりでヘルパーを作らず、共有パッケージを使う」とか、「型を推測せず、境界で検証するか型のついた SDK を使う」とか。
    • これらを定期的にスコアリングするエージェントを走らせ、小さな PR を出して高速にレビューしマージすることで、修正コストの肥大化を防ぐ

    申し送り¶

    最後に申し送りとして以下が述べられています。

    • ここまでは実ユーザ込みでうまくスケールしているが、長期的に設計の一貫性 (coherence) が維持されるかはわからない。
    • どうすれば人間の判断をよりレバレッジ効かせられるかも学習中。
    • 開発規律の中心はコードがら土台作り (scaffolding) へ移る。ツール・環境・抽象化・FBループの設計が重要。

    Hashimoto 氏の主張からの深化¶

    長くなってしまいましたが、OpenAI チームの共有は、Hashimoto 氏の主張の延長線上にあるものと理解しました。

    • Hashimoto 氏が重要視する AGENTS.md の改善とツーリングは OpenAI が主張する scafollding や leverage と矛盾しない
    • OpenAI の主張は、docs/* によるナレッジマップの最適化、 leginility 志向、リンターや CI の高度化により、この主張を無理なく深化させている

    Martin Fowlar 氏の反応¶

    OpenAI の発信から 1 週間程度たったタイミングで、Martin Fowler 氏がこれに反応しています。 記事に対して名指しで反応するあたり、OpenAI の記事が注目を集めていることが伺えます。

    以下は Martin Fowler 氏の主張です。

    記事への批判¶

    • OpenAI チームは、内部品質と保守性にのみ言及しており、プロダクトの機能や振る舞いについての検証が見当たらない。
    • OpenAI チームは、コードが AI で保守可能であるということを主張するインセンティブがある。
      • 明言してませんが「ポジショントークじゃね？」というニュアンスだと思っています。

    ハーネスは将来のサービステンプレートになるか？¶

    • 多くの組織は大抵 2 ~ 3 個の技術スタックしか持たないので、似たような技術スタックのアプリケーションを持つことになる
    • 現在のゴールデンパスのように、チームは最初に汎用ハーネスを使うようになるかもしれない。
    • ハーネスはテンプレート化して共有されうるか。既存のゴールデンパスのようにフォークや同期の難しさがあるかも。

    高い AI 自律性のためには、実行環境側の制約が必要か？¶

    • AI コーディングの盛り上がりの中には「LLM があれば実行環境/言語/パターンの制約は無い」という主張があるが、そんなことはないはず。
    • 実際に、OpenAI の事例でも「解の空間」を狭めているように見える。
    • 柔軟性の一部を捨てて、プロンプト・構造化ルール・ハーネスに置き換えている。

    技術スタックとトポロジの数は、限られたものに収束していくか？¶

    • 人間にとって良いものは AI にとっても良いが、好みやスタイルや癖レベルのものは今までよりも重要ではなくなる
    • いいハーネスがあったり、AI フレンドリーだったりするフレームワークや SDK を選ぶようになるかもしれない
    • これはコードベース構造やトポロジにも当てはまるかもしれない。ハーネス化しやすい構造があるなら、それがデフォルトになるのかも。
    • OpenAI の記事を読むと、「リポジトリ構造の厳格さ」「強制するルール」が重要なのはわかるが、それが具体的に何なのかはイメージしづらい。
    • AI 熱狂派は、自然言語が新たな抽象化レイヤになると主張しているが、実際はこのようなトポロジが新たな抽象化レイヤになるのかもしれない

    2つの未来：AI以前とAI以後のアプリ保守¶

    • AI自律性が高まった未来に、どの技術は既存アプリにも適用でき、どの技術は「最初からハーネス前提で作ったアプリ」でないと機能しないのか
    • 古いコードベースに対しては、ハーネスを後付けする価値があるかを判断する必要がある。エントロピーの大きさ故に、やる価値が無いかもしれない。

    What’s your harness today?¶

    • OpenAI の 5 ヶ月という時間は、短期成果を狙って気軽に飛び込めるものではない
    • しかし今自分にとってのハーネスを考える価値はある
      • pre-commit フック
      • カスタムリンタのアイデア
      • コードベースに課したい機械的な制約
      • ArchUnit のようなコードベース構造のテスト

    最後に¶

    • OpenAI チームのやっていることは、大量のマークダウンをメンテすることではない
    • 広範なツール、ナレッジベースのキュレーション、コードやリポジトリ設計そのもの
    • コード設計はそれ自体が巨大なコンテキストである
    • 「ハーネス」が「1プロンプトで動くLLMベースのコードレビューエージェント」と大衆に誤解されるまでなら、息を止めていられるだろう

    うーん、ちくちく言葉w 表面的なツールやプロセスに走るだけでは意味がないという主張だと思いますが、 Martin Fowler が言うと重みがあります。

    筆者の意見¶

    Hashimoto 氏、OpenAI、Fowler 氏を含む最近の周辺記事を読んでの、私の意見を書いておきます。

    開発生産性が上がるだけではプロダクトは良くならない¶

    当たり前ですが、開発生産性が上がってもプロダクトが良くなるとは限りません。

    手段が目的化してしまったプロダクトに価値はないので、開発生産性に見合うだけの、高品質なプロダクトアイデアを供給し続ける必要があると考えます。 それが不可能なのであれば、ハーネスエンジニアリングというプロセスの話ではなく、プロダクトの話に力点を置くべきでしょう。

    AI は増幅器¶

    エージェントはプロンプトを強く学習する傾向にあるが故に、AI は使用者の増幅器(アンプ)である という指摘はしばしばなされます。 この原則はハーネスエンジニアリングでも崩れることはないでしょう。

    あるいは自動化も同じでしょう。ゴミを自動化したら自動でゴミが出来ます。

    また、AI が実装を置き換える時代になったことで、我々は具体を学習する機会を失いつつあります。具体を失うということは、そこから得られる抽象的思考を失うことかもしれません。 私たちはこれを重く捉え、時には立ち止まって学習し、自動化レバレッジの元手を増やしていく必要があります。

    リンタやテストに投資する¶

    プロンプトは非決定的かつ有限である一方で、リンタやテストは決定的です。

    OpenAI チームが 500 万行という規模でスケールできたのは、非決定的な AI コンテキストの層を薄く保ったことが大きいと考えています。

    リンタやテスト、スクリプトといった周辺環境に投資し、不変条件を定量的に定義できることが、ハーネスエンジニアリング時代により必要な条件になるものと捉えています。

    振る舞いを定量に寄せる¶

    Fowler 氏も指摘されている通り、プロダクトの機能性や振る舞いは非常に重要です。しかし、プロダクトの全ての側面を定量的に計測することは不可能です。

    スコアリングや AI を用いて如何に振る舞いを自動・定量・計測可能に寄せられるかがポイントになるのではないでしょうか。

    小さくはじめる¶

    全ての適用し自己改善ループを回すことは難しいかもしれませんが、Fowler 氏が言及しているように、小さなことから部分的に改善できるかもしれません。

    • エージェントに gh を渡して、CLI 上で PR の内容をチェックできるようにする
    • AGENTS.md の一部をカスタムリンタで代替する
    • エラー分析をスキル化する
    • Chrome Devtools MCP などで UI を AI に解釈させる

    新規 PJ であっても既存 PJ であっても、それぞれの粒度でステップを踏むことができるはずです。

    AI の進化を織り込む¶

    2024 年には複雑なパイプラインが必要だった機能が、今では単一のプロンプトで動作するようになっています。

    ハーネスや技術スタックに関しても、AI の精度向上を織り込んで、何度も大きくテコ入れをする前提を持っておくべきだと思います。 また、より危殆化しない技術を選定するための審美眼も重要になるでしょう。

    さいごに¶
